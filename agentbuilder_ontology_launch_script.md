# AgentBuilder × Ontology 产品功能发布会

## 10 分钟现场演示脚本

> **产品定位**：AgentBuilder 新增「本体（Ontology）」能力，让 Agent 从"单点规则触发"进化为"跨域因果推理 + 实时对话副驾驶"
>
> **核心叙事线**：32 条规则预警 → Agent + Ontology 智能分诊只留 2 条 → 本体穿透发现"报复性解绑" → 预加载挽留策略 → 一通电话 Agent 实时副驾驶 → 完美闭环
>
> **场景**：银行私行部流失预警客户挽留
>
> **角色**：你扮演理财经理"小朱"，台下是创新奇智的客户、合作伙伴和行业观众

## 第一幕：Ontology 驱动的智能分诊（2 分钟）

### 🖥️ 画面：今日待办首页

**【操作】** 打开页面，停留 3 秒让观众看清全局。

**【口播】**

> "各位好，我们先从一个真实的业务痛点说起——银行的理财经理每天早上打开系统，面对的是**上百条规则引擎推送的预警**。哪些是真正要跟进的？哪些是误报？传统方式靠人肉翻阅，效率极低。"
>
> **（手指向顶部导航栏，朗读）**
>
> *Agent 已于昨晚 0:00 完成智能分诊，基于 Ontology 对 **32 条**原始预警进行分析，筛选出 **2 条**需要关注的事项。*
>
> "32 到 2。**这就是 AgentBuilder 新增 Ontology 能力后的核心价值**。"

**（停顿，切换到"为什么"）**

> "传统规则引擎的问题是什么？它只能看到**单一维度的信号**——'这个客户取消了代扣'，就推一条预警。但它不知道为什么取消、有没有前因后果。"
>
> "**Ontology 做的事情，是在 Agent 的推理过程中引入一张活的知识图谱**——把散落在交易系统、网点系统、客诉系统中的孤岛数据，用实体和关系串联起来。Agent 不再是'看到信号就报警'，而是'沿着图谱追溯因果，再决定要不要报警'。"

**（指向过滤说明）**

> "被过滤掉的 30 条中：18 条是账户正常波动、9 条是批量系统维护触发的、还有 3 条是 Agent 通过**家庭本体交叉验证**确认为家庭内部调整，直接自动排除了。"
>
> "而张总作为千万级 VIP，Agent 虽然判定为低风险，但出于审慎原则没有自动排除——留给理财经理最终确认。**这就是 Agent 与 Ontology 配合的'智能门槛'设计。**"
>
> "我们先看高风险这位——李总。**AUM 2,850 万，风险评分 92 分**。点进去看。"

## 第二幕：本体穿透 × Agent 思维链（2.5 分钟）

### 🖥️ 画面：李总详情页

**【口播】** —— 顺序：左侧图谱 → 右侧思考链

> "这个页面是 AgentBuilder 里 Ontology 能力最直观的呈现。"

**（一）指向左侧本体网络图**

> "左边是 Agent 在运行时为李总自动构建的**客户本体网络**。注意，这不是一个事先画好的静态标签——它是 Agent 在分诊过程中，**按需从多个数据源拉取实体、推导关系后动态生成的**。"

**【操作】** 逐一指出关键节点（用鼠标悬停触发 Tooltip）：

> ① "最上面两个黄色和蓝色节点——**授权变更**和**网点访问**，这是触发事件。Ontology 中我们把它们建模为 **Event Object**。"
>
> ② **（指向红色线）** "中间这条红色连线非常关键——**客户：李总 → 代缴授权：操作 撤销/解绑 (T-0)**。这是触发预警的源头动作。"
>
> ③ "再看下面——**投诉工单**连着**银行网点(XX支行)**，旁边还有**客服录音**作为语音证据。**这三个实体来自三个完全独立的系统**：交易系统、网点系统、客诉系统。**Ontology 的价值，就是把这些数据孤岛用标准化的实体和关系串在一起。**"
>
> ④ **（关键停顿，hover 到紫色节点，等 Tooltip 弹出）** "最后，左下角——**营销活动：高端红酒品鉴会**。李总 90 天前参加过。等一下你会看到，Agent 是怎么用这条'冷数据'来救场的。**这就是 Ontology 的长尾记忆能力。**"

**（二）转向右侧 Agent 思考链**

> "右边是 Agent 的**思维链条（Chain of Thought）**。这是 AgentBuilder 的核心能力——让 Agent 的推理过程完全透明、可审计。"
>
> "每一步下面蓝色的方框，就是 Agent 在执行这一步时**实际调用的 Ontology 查询**——"

**（逐步解读）**

> ① "**感知**——检测到代扣撤销，月均归集额从 85 万直接归零。"
>
> ② "**本体溯源**——Agent 沿着图谱，追溯到 T-3 天在 XX 支行的网点访问，停留 145 分钟，远超 30 分钟的阈值。注意这一步的查询：`【实体】客户:李总 -【关系】网点访问.地点 → XX支行`——**这就是一次典型的 Ontology Traverse 操作**。"
>
> ③ "**情绪分析**——提取投诉工单的情绪标签，置信度 0.92，工单未闭环。**Agent 甚至穿透了客服录音，匹配到了[排队、VIP、欺骗]这些关键词。**"
>
> ④ "**结论**——服务体验导致的报复性解绑。注意这里的评分拆解：`网点滞留(+25) + 投诉未闭环(+30) + 代扣批量解绑(+22) + 情绪指标(+15) → 92 分`。**每一分都有 Ontology 实体作为白盒化证据。**"

## 关键概念速查表（供演示前复习）

| 概念 | 一句话解释 | 演示中的体现 |
|------|-----------|-------------|
| **Ontology** | 用标准化的实体和关系连接数据孤岛的知识图谱 | 李总图谱中 8 个节点、10 条边来自 3 个独立系统 |
| **Event Object** | 将事件建模为一等对象，自带时间、地点等属性 | 授权变更、网点访问被建模为事件层 |
| **Ontology Traverse** | Agent 沿图谱关系链路追溯因果 | 思考链中的本体溯源步骤 |
| **Pre-loaded Strategy** | 通话前基于 Ontology 预计算的决策树 | 左侧动态决策树的全部分支 |
| **Incremental Correction** | 通话中 Agent 实时修正预设策略 | 导航仪的 LLM 动态策略修正卡片（850ms） |
| **家庭本体交叉验证** | Agent 通过配偶/家庭关系判断误报 | 张总案例中同地址双向签约确认 |
| **白盒化** | Agent 每一步推理都有可追溯的 Ontology 证据 | 思考链中蓝色方框的实体/属性查询 |
